## 1. 产品定位

### 1.1 产品愿景

智囊团是一个角色化 AI 智囊团产品，用户以"老板/主角"身份参与多个 AI 角色的沉浸式讨论。不是 AI 问答工具，而是一种新的 AI 交互范式 —— **Multi-Agent UX**。

> **核心价值主张**：用户是老板，AI 是团队。在有冲突、有人设、有节奏的讨论过程中获得多维度分析和决策辅助。

### 1.2 目标用户

- 对 AI 交互有兴趣的技术爱好者
- 需要多视角决策分析的用户
- 喜欢沉浸式角色扮演的用户

### 1.3 核心对比

| 对比维度 | ChatGPT 等 | 智囊团 |
|---------|-----------|--------|
| 交互方式 | 我问，AI 答 | 用户参与 + 旁观 + 指挥 |
| 输出形式 | 单一视角答案 | 多视角碰撞 + 收敛结论 |
| 体验感受 | 工具 | 像参加一场有趣的会议 |
| 情感连接 | 弱 | 角色人设带来记忆点和沉浸感 |

## 2. 产品原型

### 2.1 页面流程

应用分为 5 个核心页面，按线性流程串联：

| 页面 | 路由 | 功能描述 | 进入条件 |
|-----|------|---------|---------|
| 启动屏 | `/` | 加载上次配置，自动切换主题，1秒后跳转 | 冷启动 |
| 模板选择 | `/templates` | 选择讨论场景，支持分类筛选和 JSON 导入 | 启动后 |
| LLM 配置 | `/llm` | 配置模型、API Key、自定义接口、高级参数 | 选模板后 |
| 角色配置 | `/roles` | 查看/编辑/添加角色，配置人设、提示词、模型 | LLM 配置后 |
| 讨论主界面 | `/chat` | 核心体验，多角色对话流，爽点机制，用户参与 | 进入讨论后 |

### 2.2 页面规格：模板选择

**布局**：左右双栏布局
- 左栏（固定 320px）：品牌 Logo + 分类导航 + 已选信息 + 操作按钮
- 右栏（自适应滚动）：模板卡片网格（2列），底部导入区

**模板卡片**：
- 顶部色带（5px）：代表绑定主题的视觉指纹
- 图标 + 名称 + 描述 + 角色 chip + 主题色点
- 选中态：border 加粗变为 ink 色 + 右上角 ✓ 徽章
- 悬停态：translateY(-2px) + box-shadow 加深

**主题自动切换**：选中模板时，自动读取 `data-theme` 属性，切换全局主题变量。

### 2.3 页面规格：LLM 配置

**布局**：左侧导航栏（200px）+ 右侧内容区，5 个子页签

- **全局默认**：模型选择 + Temperature / MaxToken / 超时参数
- **自定义模型**：填写名称 / 模型ID / Base URL / API Key / 接口格式
- **API Keys**：按供应商填写 Key，显示连通状态
- **免费额度**：零配置启动入口，推荐免费模型列表
- **高级设置**：重试次数 / 发言间隔 / 并发数 / 流式开关 / 自定义 Headers

> **自定义模型说明**：Base URL 支持任意 OpenAI 兼容接口，包括本地 Ollama（`http://localhost:11434/v1`）、自建代理、第三方中转。

### 2.4 页面规格：角色配置

**布局**：左侧角色列表（220px）+ 右侧编辑区（滚动）

**角色字段**：

| 字段 | 类型 | 说明 |
|-----|------|------|
| 角色名称 | text input | 显示名 |
| 角色定位 | text input | 分析视角描述 |
| 头像字 | text input | 最多 2 个字 |
| 角色颜色 | color picker | 6 种预设色 |
| 是否主持人 | toggle | 开启后负责调度逻辑 |
| 性格定义 | text input | 约束 LLM 整体性格走向 |
| 说话风格 | tag cloud | 多选标签 |
| 口头禅 | text input | 用 `/` 分隔 |
| 对主公的态度 | select | 4 种预设 |
| 使用模型 | select | 跟随全局 或 单独指定 |
| Temperature | select | 跟随全局 或 单独指定 |
| 系统提示词 | textarea | 直接发给 LLM 的 system prompt |
| 发言风格预览 | display | 静态展示 2 条示例发言 |

### 2.5 页面规格：讨论主界面

**顶栏**：
- 左：场景名徽章 + 当前话题 pill
- 右：讨论状态指示器 + 角色/主题/新讨论按钮

**角色栏**：横向排列所有角色，主持人最左，用户（主公）最右。当前发言角色显示底部下划线。

**消息流**：

| 消息类型 | 视觉规格 |
|---------|---------|
| 主持人发言 | 暖灰底 + 斜体 + 左对齐 |
| 角色发言 | 角色色调背景 + 角色色边框 |
| 用户发言 | 纯黑底 + 白字 + 右对齐 |
| 打脸事件卡 | 玫瑰红左边框 |
| 站队事件卡 | 琥珀色左边框 |
| 主持人邀请卡 | 纯黑背景 + 白字 |

**输入区**：
- 顶部提示：模式标签（插话/指挥/定夺）
- 自适应高度 textarea
- 快捷指令 chips

### 2.6 主题系统

内置 4 套主题，通过 CSS 变量实现：

| 主题名 | data-theme | 背景色 | 强调色 |
|-------|-----------|--------|--------|
| 硅谷简洁 | （默认） | `#F7F6F3` | `#E8620A` |
| 水墨暗色 | `ink` | `#1A1612` | `#C8A96E` |
| 童话梦幻 | `fairy` | `#0E0818` | `#C084FC` |
| 星际科技 | `space` | `#020509` | `#00C8FF` |

## 3. 功能设计

### 3.1 讨论引擎：核心能力

讨论引擎是产品的灵魂，运行在客户端。核心是"导演逻辑"而非"调度逻辑"。

| 能力 | 描述 | 触发条件 |
|-----|------|---------|
| 识别用户意图 | 解析用户发言的方向、情绪、指令类型 | 用户每次发言后 |
| 定向调度 | 从角色列表中选 1-2 个最相关角色回应 | 用户发言/轮次推进时 |
| 主动邀请用户 | 在关键节点发出邀请卡 | 满足邀请条件时 |
| 制造冲突 | 识别可争论点，主动调度反方角色 | 出现明确分歧时 |
| 触发爽点 | 打脸/站队/投票/反转 | 满足触发条件时 |
| 控制节奏 | 限制单次发言长度，管理轮次密度 | 每轮发言生成时 |
| 话题转向 | 用户改变方向时重新组织讨论 | 用户发言偏离当前主题时 |
| 防止拍马屁 | 用户观点有问题时调度角色反驳 | 检测到全员附和时 |
| 收束总结 | 讨论达到轮次上限或用户发出结束信号时 | 轮次上限/用户指令 |

### 3.2 主持人邀请机制

**触发时机**：
- 出现明显的观点分歧
- 某角色直接质疑了用户之前的观点
- 讨论到了需要做决定的分叉点
- 出现意料之外的信息或反转
- 讨论即将收敛、需要用户最终定夺

**邀请频率**：一场讨论中 2-3 次

### 3.3 爽点机制规格

**打脸机制**：
- 触发：角色 B 的论据直接推翻了角色 A 之前的明确断言
- 展示：玫瑰红事件卡
- 频率：一场讨论 0-2 次

**站队/阵营对抗**：
- 触发：出现重大方向分歧
- 展示：琥珀色事件卡，展示两个阵营

**投票/判定**：
- 触发：某议题已争论 3+ 轮仍无定论
- 展示：主持人发起投票，角色各自表态

**随机反转事件**：
- "探子来报"：临时引入新角色带来新信息
- "叛变"：某角色突然改变立场
- "极端假设"：主持人抛出反事实命题
- 频率：一场讨论最多 1 次

### 3.4 用户参与模式

| 模式 | 触发方式 | 处理逻辑 |
|-----|---------|---------|
| 被动邀请 | 主持人在关键节点发出邀请卡 | 用户回应 → 调度 1-2 个角色响应；跳过 → 继续 |
| 主动插话 | 用户随时在输入框发言 | 主持人解析意图 → 定向调度相关角色 |
| 直接指挥 | 用户下达明确指令 | 主持人执行指令，调度指定角色 |

### 3.5 快捷指令

- 让 [角色名] 反驳他
- 发起投票
- 换个角度
- 总结一下

## 5. 验收标准

### 5.1 角色人设

- [ ] 四个角色说话风格有明显差异，盲测能分辨是谁说的
- [ ] 同一角色在多轮对话中人设保持一致，不飘移
- [ ] 全程白话文，无文言文词汇
- [ ] 角色之间有化学反应：盟友互相呼应，对手互相较劲

### 5.2 讨论引擎

- [ ] 不出现"轮流念稿"感（主持人真的在调度，不是顺序循环）
- [ ] 讨论有开场→发展→高潮→收敛的完整弧线
- [ ] 一场讨论中稳定触发 1 次以上爽点
- [ ] 连续跑 3 个不同话题不崩溃，不出现上下文混乱

### 5.3 用户参与

- [ ] 用户发言后讨论方向真实改变
- [ ] 主持人邀请在自然时机触发，不是机械定时
- [ ] "让X反驳他"等直接指挥能被正确执行
- [ ] 用户跳过邀请后讨论能继续推进，不卡住

### 5.4 稳定性

- [ ] LLM 返回格式异常时有兜底，不白屏
- [ ] 网络超时有友好提示 + 重试按钮
- [ ] API Key 错误有明确错误信息
- [ ] 连续使用 30 分钟无内存泄漏

### 5.5 关键验证指标

| 指标 | 定义 | 目标值 |
|-----|------|--------|
| "下一轮"继续率 | 用户看完一场后主动发起第二场的比率 | >30% |
| 完成率 | 用户是否看完整场讨论（>80% 消息） | >50% |
| 用户主动发言率 | 用户从旁观变为发言的比率 | >40% |
| 二次使用率 | 次日回访率 | >20% |

## 6. 已知限制

| 限制 | 原因 | 计划解决 |
|-----|------|---------|
| 无多设备同步 | 客户端优先架构 | Phase 3 加服务端同步 |
| Key 模式限制普通用户 | 需要 API Key | Phase 3 内置 LLM 路由 |
| Prompt 可被复制 | 模板 JSON 可导出 | 靠社区生态建壁垒 |
| 免费模型能力有限 | Rate limit | 动态推荐列表 + 预期管理 UI |
