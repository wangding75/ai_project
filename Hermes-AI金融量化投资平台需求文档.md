# Hermes · AI 金融量化投资平台产品需求文档

> **版本**: V1.4 | **日期**: 2026年4月 | **市场**: 中国 A 股

---

## 一、产品概述

### 1.1 产品定位

**Hermes** 是一套面向 A 股市场的全栈 AI 量化投资平台，取名自希腊神话中的商业与信使之神，寓意精准的市场信号传递与智能的交易辅助决策。平台整合深度学习模型、多因子选股引擎、实时行情监控与多策略信号评估，目标是构建从 **数据获取 → 因子计算 → AI 信号 → 策略评估 → 风控执行 → 报告复盘** 的完整投资闭环。系统定位为**决策支持工具**，提供多维度评估结果与综合参考分，最终买卖决策由用户执行，减少人为情绪干扰，提升选股胜率与资金管理效率。

### 1.2 核心目标指标

| 目标 | 描述 |
|------|------|
| 综合选股胜率 | ≥ 75%（多策略加权平均） |
| 单票最大仓位 | ≤ 25% |
| 最大组合回撤 | ≤ 15% |
| 硬性止损线 | -5% |
| 单日最大亏损 | ≤ 8% |
| 买点最高评分胜率（SS级） | ≥ 92% |

### 1.3 系统模块总览

平台由以下 **14 个子系统**构成，各模块独立运行、数据共享：

```
数据系统 ──→ 因子引擎 ──→ AI 模型体系
                              │
              情绪周期系统 ──→ 策略体系 ──→ 交易系统
                                              │
                        风控体系 ────────────→ 执行层
                                              │
         实时监控 · 回测系统 · 报告系统 · 通信系统
                                              │
                   学习系统（持续自优化）
```

## 二、数据系统

### 2.1 数据获取能力

平台需接入以下 A 股数据源，支持实时与历史两种模式：

| 数据类型 | 说明 |
|----------|------|
| 实时行情 | A 股逐笔、分钟级、盘口买卖五档数据 |
| 历史 K 线 | 日线、分钟线（1min / 5min / 15min / 30min / 60min） |
| 涨跌停数据 | 涨停、跌停、炸板记录，含封单金额 |
| 龙虎榜数据 | 游资席位、机构席位、净买入金额 |
| 财务数据 | 营收、净利润、ROE、负债率等核心财务指标 |
| 板块数据 | 概念板块、行业板块、题材归属 |
| 资金流向数据 | 主力资金、超大单、大单净流入/流出 |
| 另类数据（扩展） | 舆情数据、社交媒体情绪（后期接入） |

### 2.2 数据源架构

采用 **适配器模式（Adapter Pattern）**，统一上层接口，底层数据源可热切换：

| 层级 | 方案 |
|------|------|
| 主数据源 | Tushare Pro（日线、分钟线、财务、龙虎榜等） |
| 备用数据源 | AkShare（当 Tushare 不可用时自动切换） |
| 接口设计 | 统一 DataAdapter 接口，上层业务无感切换 |
| 故障处理 | 主源异常时自动切换至备源，同时推送告警通知 |

### 2.3 数据处理

- **数据清洗**：去极值（Winsorize）、标准化（Standardize）、行业中性化（Neutralize）
- **数据修复**：自动检测缺失、异常值，填补或标记
- **数据验证**：每次计算前校验数据完整性与时效性
- **热缓存**：Redis 热缓存高频请求数据，减少重复 API 调用，降低延迟
- **本地缓存**：热数据本地落盘，减少重复请求，支持离线回测

### 2.4 因子计算流水线（延迟优化）

采用 **异步两阶段流水线** 降低端到端延迟：

| 阶段 | 内容 | 时序 |
|------|------|------|
| 阶段一（同步） | 核心因子（信号生成所需最小集） | 优先计算，完成后立即出信号 |
| 阶段二（异步） | 全量因子（500+）、非关键指标 | 后台异步计算，不阻塞信号推送 |

其他延迟优化措施：
- 增量计算：只计算发生变化的标的，跳过无变化数据
- 并行计算：多进程/多线程并行处理不同标的或因子组
- 预计算缓存：盘前将已知数据提前计算并缓存至 Redis

## 三、因子引擎

### 3.1 因子体系

平台需构建 **500+ 多维因子**，覆盖以下六大类别：

| 因子类别 | 数量目标 | 描述 |
|----------|----------|------|
| 基础因子 | 160+ | 价格动量、成交量、技术指标等经典量化因子 |
| 微观结构因子 | 80 | 盘口压力、买卖价差、委托队列、高频订单流特征 |
| 另类数据因子 | 70 | 舆情热度、社交情绪、搜索指数等非结构化特征 |
| 产业链因子 | 100 | 上下游联动、供应链关系、产业景气度 |
| 行为金融因子 | 100 | 羊群效应、处置效应、过度反应、散户情绪 |
| 宏观因子 | 50 | 利率、汇率、信贷、PMI、流动性环境 |

### 3.2 因子计算要求

- 支持全量批量计算（每日盘后更新所有因子）
- 支持增量实时计算（盘中核心因子动态更新）
- 因子有效性定期验证：IC 值、IR 值、因子分层收益
- 因子重要性排序：通过 XGBoost 计算各因子对预测目标的贡献度，动态筛选 Top 因子

## 四、AI 模型体系

### 4.1 模型清单

平台需部署以下 AI 模型，各有独立用途，最终通过集成模型汇总决策：

| 模型 | 技术方案 | 核心用途 | 准确率目标 |
|------|----------|----------|------------|
| 选股评分模型 | XGBoost | 对候选股打分（0–100），输出买入优先级 | AUC ≥ 0.84 |
| 板块轮动模型 | Transformer + GAT 图注意力 | 预测板块强弱切换，辅助行业配置 | AUC ≥ 0.80 |
| 趋势预测模型 | LSTM + Attention | 预测个股未来 N 日趋势方向 | 准确率 ≥ 60% |
| 情感分析模型 | NLP / BERT | 解析新闻、公告、社媒文本，输出情绪分 | 准确率 ≥ 75% |
| 高频信号模型 | 高频神经网络 | 捕捉盘口微观结构信号，辅助买卖时机 | — |
| 仓位优化模型 | PPO 强化学习 | 动态调整各持仓权重，最大化风险调整收益 | 年化收益 ≥ 20% |
| 集成决策模型 | Stacking 集成 | 融合以上各模型输出，生成最终综合评分 | 准确率 ≥ 76% |

### 4.2 模型训练与更新

- **训练方法**：Walk-forward 滚动窗口（详见回测系统 §10.2）
  - 训练窗口：3 年滚动历史数据
  - 验证窗口：紧邻的 1 个季度
  - 重训频率：每季度一次
  - 严格时序分割，禁止随机打乱，防止数据泄漏
- **版本管理**：每次重训后对比新旧模型 IC、胜率，优于旧版才替换
- **在线学习**：支持实盘反馈数据持续微调（小批量增量训练）
- **因子重要性**：XGBoost 模型每次训练后输出 Top 40 核心因子，供因子引擎优先计算

## 五、策略体系

### 5.1 策略分类

平台需实现以下 **14 套选股策略**，各策略独立运行、独立输出评估结果（详见 §5.2 展示原则）：

**游资类策略**

| 策略 | 核心逻辑 |
|------|----------|
| 涨停板策略 | 识别强势涨停板；**首板/二板/三板独立评分**，各阶段评分权重与风险参数不同；次日竞价溢价买入 |
| 游资跟踪策略 | 追踪龙虎榜游资席位净买入，复制强势游资操作 |
| 龙虎榜策略 | 分析席位历史胜率，机构/游资资金博弈判断 |
| 游资战法识别 | 自动识别常见游资战法（一字板、炸板回封、换手洗盘等） |

**资金类策略**

| 策略 | 核心逻辑 |
|------|----------|
| 北向资金策略 | 跟踪沪深港通北向资金持续流入标的 |
| 融资融券策略 | 融资余额变化率、融券空头比例作为多空信号 |
| 机构抱团策略 | 识别机构持仓集中度高且季报持续加仓的标的 |
| 主力资金策略 | 超大单净流入持续性判断，过滤洗盘与对倒 |

**技术/量化类策略**

| 策略 | 核心逻辑 |
|------|----------|
| 多周期共振策略 | 日线 + 60 分钟 + 15 分钟三周期同向信号共振入场 |
| 板块轮动策略 | 依据板块强弱排名，动态切换至近期领涨板块 |
| 涨停数据分析策略 | 涨停连板数、炸板率、封单比等数据量化选股 |
| 日内做 T 策略 | 同一持仓内低吸高抛，降低持仓成本（T+0 不适用则次日） |

**宏观/跨市场类策略（扩展）**

| 策略 | 核心逻辑 |
|------|----------|
| 跨市场联动策略 | AH 溢价、港股领涨 A 股联动标的识别（后期接入港股数据） |
| 期权策略 | 利用期权对冲持仓风险（待 A 股期权市场成熟后接入） |

### 5.2 策略结果展示原则

**系统定位：决策支持工具，不是决策者。** 核心原则如下：

- **独立展示**：每只候选股的 14 套策略各自输出独立评估结果，不合并为单一黑箱信号
- **无否决权**：任何单一策略均不具备对其他策略的否决权，信号通过加权评分并行处理
- **用户决策**：系统提供各维度评估结果与综合参考分，最终买卖决策由用户执行
- **权重动态调整**：各策略权重由历史胜率动态更新，胜率高的策略权重更高
- **综合参考分**：加权汇总后输出 0–100 综合参考分，对应买点分级与仓位建议（见风控体系），供用户参考

## 六、情绪周期系统

### 6.1 四阶段情绪框架

市场情绪划分为四个阶段，对应不同的仓位策略：

| 阶段 | 市场特征 | 建议仓位 |
|------|----------|----------|
| 🧊 冰点期 | 成交量萎缩、情绪极度悲观、连续下跌 | 空仓或轻仓（≤ 20%） |
| 🌱 回暖期 | 量能温和放大、板块轮动出现、情绪修复 | 逐步建仓（20–50%） |
| 🚀 主升期 | 赚钱效应强、涨停数量多、游资活跃 | 满仓积极做多（50–90%） |
| 🔥 高潮期 | 情绪过热、成交量极度放大、跟风盘涌入 | 逐步减仓、控制风险（≤ 50%） |

### 6.2 情绪量化指标体系

| 指标 | 计算方式 |
|------|----------|
| 涨停数量 | 当日 A 股涨停家数（剔除 ST） |
| 炸板率 | 曾涨停但收盘未封板比例 |
| 连板率 | 昨日涨停今日继续涨停比例 |
| 赚钱效应指数 | 综合胜率 · 平均涨幅 · 活跃度加权 |
| 情绪分 | 0–100 综合评分，每日收盘自动计算 |

### 6.3 情绪分阈值定义

| 情绪分范围 | 阶段 | 建议仓位上限 |
|-----------|------|------------|
| 0–30 | 冰点期 | ≤ 20% |
| 31–55 | 回暖期 | 20–50% |
| 56–75 | 主升期 | 50–90% |
| 76–100 | 高潮期 | ≤ 50% |

### 6.4 阶段平滑机制

为防止情绪分在阈值附近频繁跳变导致仓位频繁切换，采用 **5日均线穿越确认机制**：

- 情绪分的 5 日移动平均线（MA5）**持续穿越阈值**才触发阶段切换
- 单日情绪分突破阈值不立即切换，需 MA5 也同向穿越确认
- 此机制过滤噪音信号，防止短期波动导致频繁升降仓

### 6.5 功能需求

- 每日收盘后自动计算情绪分，识别当前所处阶段
- 情绪拐点识别（由冰点转回暖、由高潮转回落）
- 情绪阶段与仓位映射自动化建议
- 历史情绪数据回测验证（2020–2025），统计各阶段策略胜率差异

## 七、交易系统

### 7.1 交易模式演进规划

交易系统分两个阶段交付，A 阶段优先上线，B 阶段在接口预留的基础上平滑升级，**数据结构全程保持一致**，升级时只切换数据来源，不重构业务逻辑。

| 维度 | A 阶段（辅助决策） | B 阶段（自动交易） |
|------|-------------------|-------------------|
| 定位 | 系统出信号，人工在券商 App 执行 | 系统直连券商 API，程序自动下单 |
| 下单方式 | 人工手动 | 自动下单 / 撤单 / 改单 |
| 持仓同步 | 用户手动回填成交信息 | 券商 API 成交回报自动同步 |
| 适用场景 | 上线初期、策略验证期 | 策略成熟、信号稳定后 |
| 券商依赖 | 无 | QMT / miniQMT / XTP 等交易接口 |

> **接口预留原则**：A 阶段所有交易记录的数据结构，须与 B 阶段券商 API 返回字段对齐，B 阶段上线后只需将数据写入方式从「用户填写」切换为「API 自动回填」。

---

### 7.2 多账户架构

系统支持管理多个交易账户，两种运行模式：

| 模式 | 描述 | 适用场景 |
|------|------|----------|
| **Mode A** | 同一策略、不同资金规模 | 多账户跑相同信号，仅仓位金额按比例缩放 |
| **Mode B** | 不同策略、同期运行 | 不同账户运行不同策略组合，用于 A/B 对比测试 |

- 各账户独立维护持仓、盈亏、风控状态
- 信号推送可按账户分别配置接收渠道
- 效果追踪支持跨账户策略对比分析

### 7.3 选股流程

```
每日盘后
  └─ 全量因子计算
  └─ AI 模型评分（XGBoost + 集成模型）
  └─ 14 套策略筛选
  └─ 情绪阶段判断
  └─ 生成预选股池（Top 20）
  └─ Telegram / 飞书推送候选清单

次日盘前（07:30–09:15）
  └─ 竞价异动分析
  └─ 宏观环境评分（大盘情绪 / 板块热度）
  └─ 最终买入清单确认（Top 5–10）

核心交易时段（09:15–09:35）
  └─ XGBoost 实时重新评分（结合竞价数据）
  └─ 评分 ≥ 85（S级）→ 竞价阶段买入
  └─ 09:30 开盘再次确认 → 建仓完成
```

### 7.4 买点分级体系

建议仓位为参考区间，**最终仓位须满足 §8.1 单票 ≤25% 硬上限与 §8.2 凯利公式双重约束**。

| 级别 | 综合评分 | 建议仓位（上限25%内） | 目标胜率 |
|------|----------|--------------------|----------|
| SS 级 | ≥ 90 分 | 20–25% | ≥ 92% |
| S 级 | ≥ 85 分 | 15–20% | ≥ 87% |
| A 级 | 75–85 分 | 10–15% | ≥ 82% |
| B 级 | 65–75 分 | 5–10% | ≥ 75% |
| 不入 | < 65 分 | 0% | — |

### 7.5 A 阶段：推送通知 + 人工确认回调

#### 推送消息格式

Telegram（主渠道）/ 飞书推送的买入建议需携带交互按钮，用户点击后触发回填流程：

```
📊 买入建议 | 09:24
─────────────────────
股票：XX（000001）
评分：88 分（S 级）
建议仓位：30%  |  建议金额：约 ¥30,000
触发策略：涨停板 + 龙虎榜 + XGBoost
─────────────────────
[✅ 已买入]   [❌ 未采纳]   [⏳ 待定]
```

- 点击「**已买入**」→ 弹出回填表单，用户填写实际成交信息
- 点击「**未采纳**」→ 记录放弃原因（可选），信号标记为「未执行」
- 点击「**待定**」→ 30 分钟后再次推送提醒

#### 回填表单字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 实际买入股票 | 文本 | 是 | 默认填入推送股票，可修改（如换股） |
| 实际买入数量 | 数字（股） | 是 | 用于计算实际仓位 |
| 实际成交均价 | 数字（元） | 是 | 用于计算实际买入金额与滑点 |
| 实际买入金额 | 数字（元） | 否 | 可由数量×均价自动计算 |
| 备注 | 文本 | 否 | 记录当时判断依据或调整原因 |

卖出时同样触发回填，字段包括：实际卖出数量、实际成交均价、卖出原因（止损 / 止盈 / 情绪 / 时间止损 / 其他）。

### 7.6 B 阶段：券商 API 自动同步

B 阶段上线后，7.5 的回填表单**对已接入 API 的账户自动隐藏**，成交数据由券商 API 成交回报直接写入，字段与 A 阶段完全一致：

| 字段 | A 阶段来源 | B 阶段来源 |
|------|-----------|-----------|
| 实际买入股票 | 用户填写 | API 成交回报 |
| 实际买入数量 | 用户填写 | API 成交回报 |
| 实际成交均价 | 用户填写 | API 成交回报 |
| 实际买入金额 | 用户填写 | API 成交回报 |
| 卖出信息 | 用户填写 | API 成交回报 |

**B 阶段新增能力**：自动下单、自动撤单（超时未成交）、账户资金与持仓实时同步、成交回报实时处理。

### 7.7 交易数据闭环与效果追踪

每笔信号从发出到最终平仓，全程记录以下核心数据，用于效果追踪、回测验证和模型迭代：

| 数据维度 | 字段 |
|----------|------|
| 信号侧 | 信号时间、股票代码、综合评分、买点级别、触发策略列表、建议仓位 |
| 执行侧 | 是否采纳、实际买入价、实际数量、实际金额、买入时间 |
| 持仓侧 | 持仓天数、最高浮盈、最大浮亏、卖出价、卖出时间、卖出原因 |
| 结果侧 | 实际盈亏金额、实际盈亏比例、持有周期收益率 |

#### 效果追踪指标

| 指标 | 说明 |
|------|------|
| 信号采纳率 | 用户接受建议的比例，反映信号可信度 |
| 采纳 vs 未采纳收益对比 | 验证信号的实际有效性 |
| 各评分段实际胜率 | SS / S / A / B 各级别真实胜率 vs 预测胜率 |
| 滑点分析 | 建议价 vs 实际成交价的平均偏差 |
| 策略贡献度 | 各触发策略对最终盈亏的归因分析 |

#### 模型迭代输入

- 实盘成交数据定期作为新训练样本，触发 AI 模型增量训练
- 策略胜率根据实盘结果动态更新，融合评分权重随之调整
- 因子有效性验证中加入实盘维度，与纯历史回测交叉对比

---

### 7.8 持仓管理

| 时间节点 | 操作规则 |
|----------|----------|
| 建仓后立即 | 设置 -2.3% 止损单（盘中动态跟踪） |
| 盈利达 20% | 考虑减仓 20%，锁定部分利润 |
| 盈利达 8%（尾盘） | 减仓 30%，防止隔夜回撤 |
| 14:55 | 评估持仓，决定是否持有隔夜 |
| 次日卖出策略 | 依据次日竞价高开幅度 / 情绪决定卖出节奏 |

**动态跟踪卖出规则**：以 AI 综合评分持续跟踪持仓股，**评分跌破阈值时触发卖出信号**，不设固定持仓天数上限。时间止损（超期未达预期）作为兜底规则，仅在评分持续低位且无明确催化剂时触发。

---

### 7.9 卖出规则

- **止损**：触及 -5% 硬性止损，无条件执行
- **止盈**：达到目标收益区间，分批卖出
- **情绪止盈**：情绪进入高潮期，主动降仓
- **时间止损**：持仓超过 N 日未达预期，无条件清仓
- **连续亏损保护**：连续亏损超过阈值，强制降至轻仓或空仓

## 八、风控体系

### 8.1 仓位风控规则

| 规则 | 参数 |
|------|------|
| 单票最大仓位 | 25% |
| 组合最大持仓数 | 建议 4–6 只 |
| 硬性止损线 | -5% |
| 组合最大回撤 | 15% |
| 单日最大亏损 | 8% |

### 8.2 仓位计算基准

- **仓位基数**：账户总资产的 25%（总资产 = 现金 + 持仓市值），而非单纯可用现金
- **凯利公式**：用于计算理论最优仓位；取「凯利结果」与「25% 上限」中的 **较小值** 作为实际仓位建议
- **公式**：`实际仓位 = min(Kelly公式结果, 25%)`
- **目的**：通过双重约束（Kelly 理论值 + 硬上限），防止过度集中导致回撤失控

### 8.3 触发机制

- **连续亏损强制空仓**：连续亏损 N 次或回撤触及 15%，强制清仓休息
- **情绪保护机制**：情绪进入冰点期，自动降低仓位上限至 20%
- **紧急预案**：系统异常 / 黑天鹅事件触发，一键清仓并推送告警

### 8.4 心理纪律管理

- 系统自动记录每笔操作与情绪状态，定期输出心理纪律分析报告
- 超出纪律操作（未达评分买入、止损前手动卖出）自动记录并预警
- 连续胜利后提示"注意过度自信风险"，连续亏损后提示"注意恐慌情绪"

## 十、回测系统

### 10.1 回测严格性标准

回测须满足以下严格约束，防止前向偏差（Look-ahead Bias）和过拟合：

#### 复权方式

使用**前复权**（Backward-adjusted，以当前价格为基准向历史调整）：
- 行业标准做法，因子计算与历史价格使用统一的前复权序列
- 回测完成后的实盘信号须还原为不复权价格用于实际下单

#### 财务数据时间对齐

- 严格使用**投退日对齐**：财务数据（营收、净利润、ROE 等）仅在实际公告披露日之后才对回测可见
- 禁止使用报告期（如季报截止日）作为数据可用日，防止财报前向偏差
- T 日回测只能使用 T-1 日及更早的所有数据

#### 涨停/跌停处理

| 场景 | 处理规则 |
|------|----------|
| 信号买入日为涨停日 | **不执行买入**，顺延至下一个可交易日（涨停价不可作为买入价） |
| 信号卖出日为跌停日 | **不执行卖出**，顺延至下一个可交易日 |
| 连续涨停/跌停 | 继续顺延，直至恢复正常交易 |

#### 滑点与手续费配置

| 项目 | 参数 |
|------|------|
| 买入滑点 | 0.1%（相对买入价） |
| 卖出滑点 | 0.1%（相对卖出价） |
| 双向手续费 | 0.03%（买 + 卖合计） |
| 合计成本 | 约 0.23%（单次完整交易） |

#### 流动性约束

- 单笔订单成交量 ≤ 当日成交量的 **5%**，超过则按 5% 部分成交，剩余顺延

### 10.2 模型训练方法

采用 **Walk-forward 滚动窗口**（严格时间序列分割，无随机打乱）：

| 参数 | 配置 |
|------|------|
| 训练窗口 | 3 年滚动历史数据 |
| 验证窗口 | 1 个季度（训练窗口之后的紧邻数据） |
| 重训频率 | 每季度重训一次 |
| 数据分割 | 严格时间序列顺序，禁止随机打乱 |

Walk-forward 流程：
```
[Train: T-3yr → T] → [Validate: T → T+1Q] → 上线
                                               ↓ 每季度
                     [Train: T-3yr+1Q → T+1Q] → [Validate: T+1Q → T+2Q] → 更新
```

### 10.3 功能需求

| 功能 | 描述 |
|------|------|
| 无限迭代回测 | 对策略参数自动搜索最优组合，支持多轮迭代优化 |
| 单策略回测 | 对单一策略进行独立历史验证 |
| 多策略组合回测 | 模拟多策略同时运行时的组合净值曲线 |
| 滑点模拟 | 按 10.1 配置精确模拟冲击成本 |
| 手续费计算 | 买卖双向手续费精确计入 |
| 风险指标计算 | 夏普比率、最大回撤、卡玛比率、年化收益 |
| 资金曲线生成 | 可视化净值曲线与基准（沪深 300）对比 |

### 10.4 回测评估标准

| 指标 | 合格线 |
|------|--------|
| 夏普比率 | ≥ 1.5 |
| 最大回撤 | ≤ 20% |
| 年化收益 | ≥ 20% |
| 胜率 | ≥ 65% |
| 盈亏比 | ≥ 1.5 |

## 十六、股票池定义（Universe Filter）

### 16.1 过滤规则

系统每日盘前更新候选股票池，满足以下**全部条件**的股票才进入因子计算与策略筛选流程：

| 过滤维度 | 规则 | 说明 |
|----------|------|------|
| ST / \*ST | **剔除** | 含退市风险，规则异常 |
| 新股 | 上市未满 **60** 个交易日剔除 | 新股定价规律与主板不同 |
| 科创板（688xxx） | **剔除** | T+0 规则不同、投资者门槛高 |
| 北交所（8xxxxx） | **剔除** | 流动性差、交易规则不同 |
| 流动性 | 近 20 日日均成交额 ≥ **5000 万元** | 保障可交易性，防止冲击成本过高 |
| 价格下限 | 收盘价 ≥ **1 元** | 剔除仙股 |
| 市值下限 | 流通市值 ≥ **20 亿元** | 兼顾小盘活跃股，剔除极端小市值 |
| 停牌 | 当日停牌自动跳过 | 不参与当日因子计算与信号生成 |

### 16.2 覆盖范围

- 市场：沪市主板（60xxxx）+ 深市主板（000xxx）+ 创业板（300xxx）
- 预计有效股票池规模：**3000–3500 只**（随市场变化动态更新）

### 16.3 更新机制

- **每日盘前 08:00**（N8N 触发）重新生成当日股票池
- 股票池变化记录入库（新进入 / 移出），支持历史回溯
- 回测时使用**历史截面**股票池，防止幸存者偏差

## 十七、Telegram Bot 交互设计

### 17.1 Bot 能力总览

### 17.2 指令列表

| 指令 | 说明 | 示例 |
|------|------|------|
| `/status` | 系统运行状态（各模块健康度、最后更新时间） | `/status` |
| `/positions` | 当前持仓列表（股票、成本、浮盈、评分） | `/positions` |

### 17.3 交互回调设计

信号消息的交互按钮点击后流程：

```
用户点击 [✅ 已买入]
    → Bot 发送内联键盘表单
    → 用户填写：买入均价 / 买入数量
    → Bot 确认并写入数据库
    → 推送确认消息（含仓位占比、止损价计算）

    → Bot 询问放弃原因（可选，内联键盘选择）
    → 信号标记为「未执行」，记入效果追踪

用户点击 [⏳ 待定]
    → 30 分钟后 Bot 自动重新推送提醒
    → 最多重推 2 次，超时自动标记「已过期」
```

### 17.4 安全机制

- Bot 仅响应白名单 Chat ID（`TELEGRAM_CHAT_ID` 配置）
- 所有写操作（回填、触发报告）需二次确认
- 非法请求记录日志并静默忽略

## 十八、停牌处理规范

### 18.1 当日停牌

| 场景 | 处理规则 |
|------|----------|
| 当日因子计算 | 跳过停牌股，不计算因子，不生成信号 |
| 持仓股当日停牌 | 暂停评分追踪，止损监控挂起 |
| 候选股当日停牌 | 从当日候选池移除，不推送 |

### 18.2 持仓股持续停牌

- 停牌 **≥ 5 个交易日**：推送 Telegram 告警，提示关注停牌风险
- 停牌期间持仓计入总资产（按最后收盘价估值）
- 复牌当日：自动恢复评分追踪，重新纳入止损监控

### 18.3 回测中的停牌处理

- 买入信号日为停牌日：不执行买入，顺延至复牌日（同涨停处理规则）
- 持仓期间停牌：计入持仓天数，不计为可操作日

## 十九、系统故障恢复规范

### 19.1 定时任务失败处理

所有 N8N 工作流失败时执行统一恢复流程：

```
任务执行失败
    → 自动重试（间隔 5 分钟，最多 3 次）
    → 3 次均失败
        → 推送 Telegram 告警（任务名称、失败原因、失败时间）
        → 记录失败日志（task_name, failed_at, error_msg, retry_count）
        → 当日跳过，标记为「待补跑」
    → 人工确认后可手动触发补跑
```

### 19.2 各任务失败的影响与处理

| 失败任务 | 影响 | 应对 |
|----------|------|------|
| 夜间数据拉取失败 | 因子计算缺数据 | 告警 + 次日补拉，当日回测跳过该日 |
| 夜间因子计算失败 | 无法生成信号 | 告警 + 使用前一日因子值（降级），标记信号为「低可信度」 |
| AI 评分失败 | 无候选股池 | 告警 + 跳过当日推送，不强行出信号 |
| 盘前推送失败 | 用户收不到信号 | 自动重试，失败则记录，用户可用 `/report` 手动触发 |
| 盘中监控失败 | 止损监控中断 | 立即告警（高优先级），人工介入 |

### 19.3 数据一致性保障

- 所有写数据库操作使用事务，失败回滚
- 每日收盘后自动校验当日数据完整性（信号数、因子数、行情数）
- 校验不通过推送告警，记录缺失明细

## 附录 A：全局设计原则

以下原则适用于所有模块，作为架构决策的最高约束：

### A.1 决策支持定位

**Hermes 是决策支持工具，不是自动决策系统。**

| 原则 | 说明 |
|------|------|
| 独立展示 | 所有策略输出各自的独立评估结果，不合并为单一信号 |
| 无否决权 | 任何单一策略均不能阻断或覆盖其他策略的展示 |
| 用户主权 | 系统提供多维评估结果与综合参考分，买卖决策由用户执行 |
| 透明可解释 | 每个评分须附带触发策略列表和关键因子，用户可理解依据 |

### A.2 数据严格性

| 原则 | 实现要求 |
|------|----------|
| 前向偏差防控 | T 日只能使用 T-1 及更早数据 |
| 财报投退日对齐 | 财务数据以公告披露日为准，非报告截止日 |
| 复权一致性 | 回测与因子计算统一使用前复权数据 |

### A.3 系统可靠性

| 原则 | 实现要求 |
|------|----------|
| 数据源冗余 | Tushare 主源 + AkShare 备源，自动切换 + 告警 |
| 信号可追溯 | 每个信号记录完整触发链路，支持事后复盘 |
| 多账户支持 | Mode A（同策略不同资金规模）+ Mode B（不同策略 A/B 测试） |
