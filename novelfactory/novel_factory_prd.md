## 1. 产品概述

### 1.1 背景

网文创作高度依赖个人经验，单本书成败难以预测，失败成本高，经验无法复用。Novel Factory 的目标是用 AI 工厂化生产网文内容，以投资组合思维并行管理多本书，通过数据驱动淘汰低质量作品、加仓高潜力作品，概率性产出爆款。

### 1.2 核心逻辑

> 单本书 = 赌博  
> 多作品并行 + AI生产 + 数据筛选 = 可控收益系统

### 1.3 目标平台

- **Phase 1**：番茄小说（单平台）
- **Phase 2**：番茄小说（主）+ 七猫小说（同步）

### 1.4 阶段定义

| 阶段 | 目标 | 并行书数 | 自动化程度 |
|---|---|---|---|
| Phase 0 | 验证AI内容能否在平台存活（纯手动，无系统） | 3本 | 0% |
| Phase 1 | 单书最小闭环，验证系统可以持续日更并维持留存 | 3本 | 半自动 |
| Phase 2 | 多书并行，全面自动化，Portfolio 管理 | 10本+ | 高度自动化 |

> **本文档覆盖 Phase 1 和 Phase 2。Phase 0 见独立行动方案文档。**

## 2. 用户角色

系统只有一个主要用户角色。

### 运营者

**是谁**：产品的所有者和日常操作者

**核心职责**：
- 审稿：在章节发布前查看 AI 生成内容，决定通过或打回
- 发布确认：半自动填充后人工点击确认发布
- 数据录入：将平台后台的数据手动录入系统
- 决策监督：查看系统的淘汰/加仓决策，必要时手动覆盖

**不负责**：
- 写作内容（由 AI 完成）
- 数据分析（由系统完成）
- 选题决策（由系统提案，运营者确认）

## 4. Phase 1 功能需求

### F1 选题与世界观

#### F1-1 选题提案

**描述**：系统根据预设的题材方向，生成选题提案供运营者确认。Phase 1 不接入实时市场数据，题材方向由运营者配置。

**功能点**：

| ID | 功能点 | 描述 | 优先级 |
|---|---|---|---|
| F1-1-1 | 题材配置 | 运营者在设置页面配置题材方向（如末世/都市系统/修仙）、参考书目、核心爽点方向 | P0 |
| F1-1-2 | 选题生成 | 系统根据配置调用 AI 生成 3 个候选选题方案，每个方案包含：书名备选×3、一句话简介、核心金手指设定、主角人设方向 | P0 |
| F1-1-3 | 选题确认 | 运营者选择一个方案，或手动修改后确认，确认后进入世界观生成 | P0 |

**输入**：运营者填写的题材配置表单  
**输出**：选题方案卡片，含书名/简介/核心设定  
**交互规则**：
- 运营者可以要求重新生成，最多重新生成 3 次
- 确认后书名不可修改（对应番茄平台签约后书名锁定规则）

---

#### F1-2 世界观生成

**描述**：基于确认的选题，AI 生成详细的世界观设定，作为 Layer1 静态知识库的核心内容。

**功能点**：

| ID | 功能点 | 描述 | 优先级 |
|---|---|---|---|
| F1-2-1 | 世界观生成 | AI 根据选题生成：背景设定、力量体系、势力分布、核心规则、禁止项清单 | P0 |
| F1-2-2 | 世界观查看 | 运营者可在详情页查看完整世界观内容 | P0 |
| F1-2-3 | 世界观编辑 | 运营者可对世界观内容进行修改，修改后系统更新 Layer1 知识库 | P1 |
| F1-2-4 | 禁止项管理 | 单独展示禁止项清单，运营者可增删，禁止项在所有章节生成中强制约束 AI | P0 |

**输入**：确认的选题方案  
**输出**：结构化世界观文档（存入数据库作为 Layer1）

---

#### F1-3 人物设计

**描述**：AI 生成主要人物设定，包括主角、核心配角、主要反派。

**功能点**：

| ID | 功能点 | 描述 | 优先级 |
|---|---|---|---|
| F1-3-1 | 人物生成 | AI 生成主角、2-3 个核心配角、1 个主要反派的完整设定（背景/性格/动机/成长弧/关系） | P0 |
| F1-3-2 | 人物列表 | 展示所有已设计人物的卡片列表，含角色、姓名、简要说明 | P0 |
| F1-3-3 | 人物详情 | 查看单个人物的完整设定 | P0 |
| F1-3-4 | 人物编辑 | 运营者可修改人物设定，修改后同步更新 Layer1 | P1 |
| F1-3-5 | 新增人物 | 运营者触发，AI 生成新角色并加入人物库 | P1 |

### F2 大纲管理

#### F2-1 全书大纲生成

**描述**：AI 根据世界观和人物设定，生成覆盖全书的弧线大纲，以"情绪债务借贷周期"为底层逻辑（先制造情绪债务，延迟偿还，最后超额偿还产生爽感）。

**功能点**：

| ID | 功能点 | 描述 | 优先级 |
|---|---|---|---|
| F2-1-1 | 弧线大纲生成 | AI 生成 3-5 个弧线的大纲，每个弧线包含：弧线名称、章节范围、核心冲突、情绪债务设计、高潮节点、结尾钩子 | P0 |
| F2-1-2 | 大纲查看 | 以时间轴或列表形式展示全书弧线结构 | P0 |
| F2-1-3 | 弧线编辑 | 运营者可修改弧线内容 | P1 |

---

#### F2-2 章节级大纲

**描述**：在章节生成前，AI 将弧线大纲细化为章节级场景脚本，作为写作蓝图。

**功能点**：

| ID | 功能点 | 描述 | 优先级 |
|---|---|---|---|
| F2-2-1 | 章节脚本生成 | 每章写作前，AI 生成本章场景脚本：主要事件、爽点位置、章末钩子设计 | P0 |
| F2-2-2 | 脚本查看 | 运营者可在章节详情页查看本章脚本 | P0 |
| F2-2-3 | 脚本修改 | 运营者可修改脚本内容后再触发正文生成 | P1 |

---

### F3 章节生成

#### F3-1 正文生成

**描述**：核心功能。AI 基于章节脚本和三层记忆上下文生成章节正文。

**功能点**：

| ID | 功能点 | 描述 | 优先级 |
|---|---|---|---|
| F3-1-1 | 自动触发生成 | 系统每日定时自动触发当日待生成章节的写作任务 | P0 |
| F3-1-2 | 手动触发生成 | 运营者可在任意时间手动触发单章生成 | P0 |
| F3-1-3 | 生成进度展示 | 章节生成过程中展示进度状态（排队中/生成中/已完成/失败） | P0 |
| F3-1-4 | 字数控制 | 每章生成目标字数可配置，默认 2000-2200 字 | P0 |
| F3-1-5 | 生成失败重试 | 生成失败时自动重试，最多 3 次，超过后标记为失败并通知运营者 | P0 |
| F3-1-6 | 附加指令 | 运营者可在触发生成时附加文字说明（如"本章需要有重大反转"），系统将其注入章节 Prompt | P1 |

**生成规范**（固化在 Prompt 中，技术实现参考）：
- 每段不超过 80 字，手机阅读友好
- 开头 300 字内必须有冲突或行动
- 第 1000 字附近安排一个爽点
- 结尾 100 字制造下一章期待感
- 禁止超过 3 句连续心理独白

---

#### F3-2 审稿

**描述**：章节生成完成后，运营者查看内容，决定通过、打回或修改后通过。这是运营者在发布流程中的核心介入点。

**功能点**：

| ID | 功能点 | 描述 | 优先级 |
|---|---|---|---|
| F3-2-1 | 审稿通知 | 章节生成完成后，系统通知运营者（页面提示或消息通知）待审稿章节数量 | P0 |
| F3-2-2 | 审稿界面 | 展示章节正文，支持全文阅读，右侧或底部有操作按钮 | P0 |
| F3-2-3 | 通过 | 运营者点击"通过"，章节进入发布队列 | P0 |
| F3-2-4 | 打回重写 | 运营者点击"打回"并填写原因，系统重新生成（可附加指令），重新生成后再次进入审稿 | P0 |
| F3-2-5 | 直接编辑 | 运营者可直接在审稿界面编辑正文内容（用于小范围润色），编辑完成后点击"通过" | P1 |
| F3-2-6 | AI 质检报告 | 章节生成后 AI 自动输出质检报告（逻辑一致性/节奏/爽点密度/钩子强度评分），展示在审稿界面供参考 | P1 |
| F3-2-7 | 批量审稿 | 支持在章节列表页查看所有待审稿章节，快速切换 | P1 |

**状态流转**：
```
生成中 → 待审稿 → 通过（进入发布队列）
                 → 打回（重新生成）
                 → 编辑后通过（进入发布队列）
```

### F4 记忆系统

**描述**：保证章节间的逻辑连贯性，无需技术背景理解细节，但需要知道系统在做什么。

#### F4-1 三层记忆维护

**功能点**：

| ID | 功能点 | 描述 | 优先级 |
|---|---|---|---|
| F4-1-1 | Layer1 自动维护 | 世界观和人物设定变化时自动更新 Layer1 静态知识库，无需人工介入 | P0 |
| F4-1-2 | Layer2 自动更新 | 每章审稿通过后，系统自动更新 Layer2 动态状态摘要（主角状态/活跃悬念/弧线进度/上章钩子） | P0 |
| F4-1-3 | Layer2 查看 | 运营者可查看当前 Layer2 摘要内容，了解系统"记住"了什么 | P1 |
| F4-1-4 | Layer2 防漂移重建 | 每 30 章系统自动从原文重建 Layer2，防止摘要逐渐偏离真实剧情 | P0 |
| F4-1-5 | Layer3 自动读取 | 每次生成前自动读取最近 3-5 章原文作为语感上下文，无需人工操作 | P0 |

---

#### F4-2 一致性检查

**功能点**：

| ID | 功能点 | 描述 | 优先级 |
|---|---|---|---|

| F4-2-3 | 问题处理 | 运营者查看问题后可选择：忽略、手动修正 Layer1/Layer2、触发 AI 修复建议 | P1 |

### F5 发布管理

#### F5-1 发布队列

**描述**：管理待发布章节的排期，支持运营者调整发布时间。

**功能点**：

| ID | 功能点 | 描述 | 优先级 |
|---|---|---|---|
| F5-1-1 | 发布队列展示 | 展示所有已通过审稿、待发布的章节列表，含书名、章节号、计划发布时间 | P0 |
| F5-1-2 | 发布时间配置 | 运营者可配置每本书的发布时间（如每天晚上 9 点），系统自动排期 | P0 |
| F5-1-3 | 手动调整排期 | 运营者可手动修改某章的计划发布时间 | P1 |
| F5-1-4 | 日更频率配置 | 每本书可配置每日发布章节数（默认 1 章，可调整为 2 章） | P0 |

---

#### F5-2 半自动发布

**描述**：Phase 1 的发布方式。系统准备好发布内容（脚本自动填充），运营者完成最终确认和提交。

**功能点**：

| ID | 功能点 | 描述 | 优先级 |
|---|---|---|---|
| F5-2-1 | 发布提醒 | 到达计划发布时间时，系统在页面显示提醒，告知运营者有章节待发布 | P0 |
| F5-2-2 | 发布预览 | 运营者点击发布提醒后，展示本次发布的章节标题和正文内容，供最终确认 | P0 |
| F5-2-3 | 一键脚本填充 | 运营者点击"发布"按钮，系统自动打开番茄作者后台对应页面并填充标题和正文（通过脚本实现） | P0 |
| F5-2-4 | 人工提交 | 脚本填充完成后，运营者在平台页面手动点击"发布"按钮完成最终提交 | P0 |
| F5-2-5 | 发布确认回填 | 运营者发布成功后，在系统内点击"已发布"，系统更新该章节状态为已发布并记录发布时间 | P0 |
| F5-2-6 | 发布失败标记 | 运营者可标记发布失败并填写原因，系统将该章节重新放入发布队列 | P1 |

**交互流程**：
```
系统提醒 → 运营者点击查看 → 预览章节内容
→ 点击"一键填充" → 跳转番茄后台（已填充内容）
→ 运营者检查并点击平台发布按钮
→ 回到系统点击"已发布"确认
```

### F6 数据录入

**描述**：Phase 1 不自动采集数据，由运营者每日手动将番茄后台的数据录入系统。

#### F6-1 数据录入界面

**功能点**：

| ID | 功能点 | 描述 | 优先级 |
|---|---|---|---|
| F6-1-1 | 书级数据录入 | 运营者每日录入：日阅读量、加书架数、7日留存率（需进入番茄后台查看后手动填写） | P0 |
| F6-1-2 | 章节数据录入 | 运营者录入每章的完读率（番茄后台每日更新，需手动录入） | P0 |
| F6-1-3 | 快速录入模式 | 一次性录入多本书同一天的数据，减少操作步骤 | P1 |
| F6-1-4 | 数据录入提醒 | 每天固定时间（如上午 8 点）提醒运营者录入昨日数据 | P1 |

**录入字段**（书级，每日）：
- 日阅读量
- 累计阅读量
- 加书架数

**录入字段**（章节级，可按需录入）：
- 章节号
- 完读率
- 评论数

### F7 数据看板

**描述**：展示所有书的健康状况和关键指标，帮助运营者判断哪本书需要干预。

#### F7-1 书籍健康看板

**功能点**：

| ID | 功能点 | 描述 | 优先级 |
|---|---|---|---|
| F7-1-1 | 书籍列表卡片 | 每本书展示一张卡片：书名、当前状态、关键指标（留存率/完读率/书架数）、趋势箭头（上升/下降/平稳） | P0 |
| F7-1-2 | 健康状态标色 | 根据留存率自动标色：红（<25%）/ 黄（25-40%）/ 绿（>40%） | P0 |
| F7-1-3 | 关键指标历史趋势图 | 查看单本书的留存率、完读率折线图，时间范围可选（7天/14天/全部） | P0 |
| F7-1-4 | 章节完读率图 | 展示各章节完读率柱状图，直观看出哪章读者流失 | P0 |
| F7-1-5 | 系统决策建议 | 基于阈值规则（留存<25%→停更建议，>40%→加仓建议），系统在看板上展示建议标签，运营者自行决策是否执行 | P1 |

#### F7-2 生产效率看板

**功能点**：

## 5. Phase 2 功能需求（待细化）

> **说明**：以下为 Phase 2 的功能方向，在 Phase 1 验证通过后进行详细需求设计。当前仅描述功能目标和关键边界，不含具体功能点。

### F9 Portfolio 管理（待细化）

**目标**：支持 10 本以上书籍并行管理，系统自动根据数据指标进行资源分配，包括自动淘汰低留存书、自动加仓高留存书。

**关键边界**：
- 并行书数上限：可配置，目标支持 10-50 本
- 决策规则：留存率 < 25% 系统建议停更，> 40% 系统建议加仓，25-40% 建议优化
- 人工覆盖：所有系统决策运营者均可手动覆盖

---

### F10 浏览器插件（待细化）

**目标**：替代 Phase 1 的手动数据录入和半自动发布，实现数据自动采集和发布辅助自动化。

**关键边界**：
- 数据采集：通过拦截番茄/七猫后台 XHR 请求，自动获取章节完读率、留存率、段评
- 发布辅助：自动打开发布页并填充内容，保留人工确认步骤
- 平台支持：番茄小说（优先），七猫小说（后续）

---

### F11 策略决策引擎（待细化）

**目标**：将 Phase 1 中"系统展示建议、运营者决策"升级为"系统自动执行决策、运营者监督"。

**关键边界**：
- C2 StrategyAgent 每日自动运行，输出 keep/optimize/suspend/promote 决策
- 决策自动执行：停更自动暂停生成任务，加仓自动提高日更频率
- 异常情况（CRITICAL 级）仍需人工介入

---

### F12 上瘾机制设计（待细化）

**目标**：在新书启动时，AI 自动定义每本书的"上瘾画布"（核心情绪钩子/情绪燃料/读者幻想/压抑-爆发节律），作为所有创作 Agent 的隐性约束。

---

### F13 合成读者系统（待细化）

**目标**：在章节发布前，用 AI 模拟 3-5 类典型读者的反应，预测完读率和追读意愿，替代真实平台数据参与章节质量评估，解决冷启动期数据滞后问题。

---

### F14 多平台发布（待细化）

**目标**：同一本书同步发布到番茄小说和七猫小说，数据分平台采集和对比。

## 6. 非功能需求

### 6.1 性能

| 需求 | 指标 |
|---|---|
| 章节生成时长 | 单章生成完成时间 ≤ 3 分钟（正常网络下） |
| 页面响应 | 所有页面操作响应时间 ≤ 2 秒 |
| 并发支持 | Phase 1 支持 3 本书同时生成任务并发执行，互不阻塞 |

### 6.2 稳定性

| 需求 | 说明 |
|---|---|
| 生成失败自动重试 | 单章生成失败自动重试 3 次，超过后通知运营者 |
| LLM 故障切换 | 主模型不可用时自动切换 fallback 模型 |
| 任务断点续跑 | 系统重启后未完成的生成任务自动恢复 |

### 6.3 数据安全

| 需求 | 说明 |
|---|---|
| 内容本地存储 | 所有生成的章节内容存储在自有数据库，不依赖第三方 |
| API Key 加密 | 各厂商 API Key 加密存储，不明文展示 |

### 6.4 可维护性

| 需求 | 说明 |
|---|---|
| Prompt 可修改 | 所有 Agent 的 Prompt 模板支持在配置文件中修改，不需要改代码 |
| 模型可切换 | 切换模型只需修改配置文件，不需要改代码 |
| 生成日志 | 记录每次 LLM 调用的输入/输出/耗时/token 消耗，用于调试 |

## 7. 排除项

以下功能明确不在本产品范围内：

| 排除项 | 说明 |
|---|---|
| 读者端产品 | 不做面向读者的阅读界面 |
| 多用户/团队协作 | 系统只支持单一运营者，不做权限管理和多账号 |
| 自动化财务结算 | 平台收益数据只做展示参考，不做自动提现或财务管理 |
| 版权管理 | 不提供版权登记、侵权监测等功能 |
| 内容合规审核 | 不做平台内容规范的自动审核，由运营者在审稿时人工把关 |
| 自动上传封面 | 封面由运营者自行准备和上传到平台 |
