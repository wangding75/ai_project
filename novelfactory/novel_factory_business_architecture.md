# Novel Factory — 业务架构文档

> **版本**: v1.0  
> **说明**: 本文档描述业务架构中的分层、组件职责与 Agent 职责，不含具体接口、数据库设计和实现细节。

---

## 一、业务目标

### 1.1 核心问题

网文创作高度依赖个人经验和运气，存在三个根本性问题：

- **单本赌博**：一本书的成败无法预测，失败成本高，经验难以复用
- **创作效率低**：人工写作难以支撑高频更新和多书并行
- **缺乏数据反馈**：创作决策依赖直觉，无法基于读者行为做精准调整

### 1.2 解决思路

用**工业化 + 投资组合**的思维重构网文生产：

> 单本书 = 赌博  
> 多作品并行 + AI生产 + 数据筛选 = 可控收益系统

核心逻辑是：同时运行多本书，让数据说话，用留存率快速筛选潜力作品，对好书加仓、差书止损，持续试错迭代，概率性产出爆款。

### 1.3 业务目标

**直接目标**
- 实现10本以上小说并行生产，每本日更4000字以上
- 冷启动期（前20章）完成留存率验证，数据差的书快速止损
- 自动化覆盖从选题到发布的全流程，人工介入控制在每天10分钟以内

**质量目标**
- 平台章节完读率 > 35%（达到平台安全线，避免限流）
- 冷启动留存率 > 25%（系统内部淘汰阈值）
- 至少1本书留存率 > 40%（判定为潜力书，加仓运营）

**效率目标**
- 10本50万字小说的综合 API 成本控制在1200元人民币以内
- 每日人工操作时间 < 10分钟（主要用于发布确认）
- 系统可7×24小时无人值守运行，异常时通知人工介入

### 1.4 成功标准

系统上线后3个月内：
- 累计生产 ≥ 10本小说，每本 ≥ 20万字
- 自动淘汰率 ≥ 60%（系统能有效识别并停止低质量书）
- 至少1本书进入平台增长期（留存率持续 > 40%，超过50章）
- 人均管理书籍数量 > 5本（对比人工写作的1本）

### 1.5 目标平台

**MVP**：番茄小说（字节跳动旗下，免费阅读模式，机器审核，门槛低）  
**完整方案**：番茄小说（主）+ 七猫小说（同步），双平台分散风险，数据交叉验证

## 三、各层职责

### 外部平台层
对接番茄小说、七猫小说的作者后台。是内容发布的目标平台，也是读者行为数据的来源。系统不直接调用平台 API（平台未开放），所有交互通过浏览器插件完成。

### 运营接入层（浏览器插件）
作为系统与外部平台之间的唯一桥梁。承担两个核心职责：将系统生成的章节发布到平台；将平台产生的读者数据采集回系统。详见第四章组件说明。

### 业务服务层
系统的业务核心，负责管理作品生命周期、调度 Agent 任务、处理数据分析结果并执行策略决策。不直接处理内容生成，只做调度和管理。

### Agent 执行层
所有内容生成、质量评审、市场分析、运营决策的实际执行者。各 Agent 职责单一，通过流水线和事件协议协作。详见第三章 Agent 说明。

### LLM 路由层
所有 Agent 调用大模型的统一入口。屏蔽底层模型差异，支持按系统级、Agent 级、任务级三层优先级动态路由到不同模型和厂商。

## 四、Agent 体系

### 3.1 完整方案（32个）/ MVP（19个）

MVP 暂缓的 13 个 Agent 在各组中标注【暂缓】。

---

### A 组：市场情报组

**职责**：在新书启动前提供市场依据，以及在系统运行期间持续积累跨书经验。

| Agent | 职责 | MVP |
|---|---|---|
| A1 MarketScanner | 定时抓取平台热榜，分析题材趋势和增长数据，输出结构化热门题材列表 | ✅ |
| A2 CompetitorAnalyst | 拆解指定热门作品的结构，提取开局设计、爽点分布、升级节奏等套路，输出标准化"爆款套路卡" | ✅ |
| A3 TopicSelector | 综合市场趋势和竞品数据，感知当前 Portfolio 已有题材分布（避免自我竞争），生成候选题材方案 | ✅ |
| A4 KnowledgeDistillAgent | 定期汇总所有书的数据结果，提炼高留存书的共同特征和失败书的共同问题，更新全局最佳实践库供其他 Agent 引用 | 【暂缓】 |

**触发关系**：新书启动时 A3 → A2 → A1 顺序触发；A4 每月定期运行。

---

### B 组：单书创作组

**职责**：负责单本书从世界观设计到章节正文生成的完整创作流水线，以及记忆维护和质量保障。

#### 前期规划（一次性，新书启动时运行）

| Agent | 职责 | MVP |
|---|---|---|
| B1 WorldAgent | 构建世界观设定、力量体系、势力地图，生成"禁止项清单"（约束后续所有 Agent 的边界），写入 Layer1 静态知识库 | ✅ |
| B2 CharacterAgent | 设计主角、配角、反派的背景、动机、成长弧和关系网，写入 Layer1 静态知识库 | ✅ |
| B2b CharacterEvolutionAgent | 每个弧线结束后更新人物心理状态、关系强度变化和读者热度，为后续弧线的剧情规划提供人物依据 | 【暂缓】 |
| B3 OutlineAgent | 规划全书弧线结构和章节节奏，以"情绪债务借贷周期"为底层逻辑设计大纲（先制造情绪债务，再延迟偿还，最后超额偿还产生爽感） | ✅ |
| B3b SatisfactionCurveAgent | 每10章回顾近期爽点密度曲线，检测"铺垫过长"或"爽点疲劳"，并为下10章输出全局爽点分配建议 | 【暂缓】 |

#### 章节生产循环（每章触发）

| Agent | 职责 | MVP |
|---|---|---|
| B5b ContextAssemblerAgent | 在每章写作前，根据本章类型（高潮/日常/转折/新角色等）动态调整三层记忆的配比，在固定 token 预算内装配最优上下文 | ✅ |
| B4 PlotAgent | 将大纲条目扩展为具体场景设计，插入爽点/冲突/对话节点，生成3个风格版本（稳健/激进/实验）的场景脚本 | ✅ |
| B5 WritingAgent | 基于场景脚本和三层记忆上下文生成正文，同样生成3个风格版本 | ✅ |
| B6 ReviewAgent | 对 B4/B5 的多个版本进行投票评审，从内容质量、读者反应（MVP阶段由B6模拟）、爽点节奏三个维度打分，按得分决定采用/合并/打回/升级 | ✅ |
| B7 SummaryAgent | 每章审核通过后更新 Layer2 动态状态摘要，维护主角状态、活跃悬念、弧线进度、上章钩子等信息；每30章触发从原文完整重建一次以防漂移 | ✅ |

#### 质量保障（定期触发）

| Agent | 职责 | MVP |
|---|---|---|
| B8 ConsistencyAgent | 每10章运行一次，跨章节检查设定矛盾（如人物能力前后不符、世界规则被破坏），发现问题时发布 RevisionRequest | ✅ |
| B9 PlotAdjustAgent | 当追读率连续下滑或 C2 输出 optimize 决策时触发，在现有大纲框架内微调节奏，压缩铺垫或提前爽点 | 【暂缓】 |
| B10 OpeningOptimizeAgent | 当冷启动期留存低但题材本身没问题时触发，专门针对前5章做重构，对比同题材高留存作品的开局结构进行手术式优化 | ✅ |

---

### C 组：运营决策组

**职责**：每日基于平台数据做出 Portfolio 层面的资源分配决策，驱动创作侧的调整。

| Agent | 职责 | MVP |
|---|---|---|
| C1 PerformanceAnalyst | 汇总插件采集的书级留存率、章节完读率、段评情感分析结果，计算趋势（上升/平稳/下滑），输出每本书的健康报告，不做决策 | ✅ |
| C2 StrategyAgent | 接收 C1 健康报告，结合 Portfolio 整体资源状况和历史决策，对每本书输出 keep/optimize/suspend/promote 决策，触发对应的后续动作 | ✅ |
| C3 RestartAgent | 当 C2 决定 suspend 时触发，诊断失败原因（题材/开局/节奏），判断是换题材重开还是同题材换开局，生成新书提案并启动新的 A3→B1 流程 | ✅ |

**决策路由**：
```
C2 输出 promote  → 修改该书的日更频率
C2 输出 optimize → 开局问题 → 触发 B10
                   中期节奏 → 触发 B9（暂缓）
C2 输出 suspend  → 触发 C3 诊断重开
C2 输出 keep     → 无动作
```

|---|---|---|

### S 组：即时信号组

| Agent | 职责 | MVP |
|---|---|---|
| S1 ChapterHealthScorer | 章节发布前，基于文本特征（开头钩子强度、爽点密度、结尾钩子、句长节奏等）计算预测健康度，辅助 B6 做发布前质量判断 | 【暂缓】 |
| S2 DropoutDetector | 章节发布后，基于平台采集的章节完读率，检测环比骤降异常，生成流失假设（节奏慢/爽点不足/钩子弱），驱动 RevisionRequest | 【暂缓】 |

|---|---|---|

| R2 ConflictResolver | 当同一本书同时存在多个 RevisionRequest 且目标互相冲突时，进行仲裁，输出优先级排序和合并修复方案 | 【暂缓】 |

| Agent | 职责 | MVP |
|---|---|---|

## 五、核心组件职责

### 4.1 浏览器插件（Chrome Extension）

作为系统与外部平台之间的唯一物理桥梁，承担双向数据流转。

**发布助手模块**
- 定时轮询系统后端的发布队列
- 发现待发布章节时推送通知给操作人员
- 自动打开平台发布页并填充标题和正文
- 等待人工确认后执行发布（保留人工最终把关）
- 发布成功后回调系统标记状态

**数据采集模块**
- 每日定时打开平台数据后台，静默采集所有书的书级留存率、日流量等数据
- 拦截平台后台的 XHR 请求，获取章节级完读率（结构化数据，比解析 HTML 更稳定）
- 采集章节段评内容，在发布后6小时和24小时各采集一次
- 将所有采集数据推送给系统后端

**适用平台**：MVP 支持番茄小说，完整方案扩展支持七猫小说。

---

### 4.2 三层记忆系统

解决长篇创作中上下文成本随章节数线性增长的问题，使每章写作的 token 消耗保持恒定。

**Layer1 静态知识库**
- 内容：世界观设定、力量体系、禁止项清单、人物档案
- 维护者：B1 WorldAgent、B2 CharacterAgent
- 特点：全局共享，极少变动，缓存命中率高

**Layer2 动态状态摘要**
- 内容：主角当前状态、活跃悬念列表、当前弧线进度、上章结尾钩子、人物关系最新变化
- 维护者：B7 SummaryAgent（每章更新），每30章从原文重建防止漂移
- 特点：滚动更新，反映最新创作状态

**Layer3 最近原文窗口**
- 内容：最近3-5章的完整正文
- 维护者：系统自动从数据库读取
- 特点：保持语感、节奏和细节的连续性

---

### 4.3 RevisionRequest 协议

将系统从单向流水线改造为可自我修复的网状结构。任何 Agent 发现问题时，通过统一协议向上游发起修订请求，而非直接调用或报错。

**协议内容**：发起方、目标方、严重程度（minor/major/critical）、影响范围（chapter/arc/global）、问题描述、建议修复方向、受影响的章节和实体。

**处理规则**：
- minor：异步处理，不阻塞当前写作
- major：暂停当前弧线，优先修复
- critical：立即暂停全书写作，通知人工确认

---

### 4.4 多版本生成与投票机制

用于 B4 PlotAgent 和 B5 WritingAgent，通过并行生成多版本再投票选优，提升单章输出质量的稳定性。

**三个版本定义**：
- 稳健型：严格按大纲执行，适合验证期
- 激进型：在框架内最大化爽点密度，适合留存下滑时
- 实验型：保留主干，在某个细节上做意外处理，适合增长期寻找新爆点

**投票方**：B6 ReviewAgent（内容质量）+ SR2 ReaderSimulator（读者反应，MVP阶段由B6代替）+ B3b SatisfactionCurve（爽点节奏符合度）

**裁决规则**：高分直接采用；中分采用并附改进建议；低分合并重写生成第4版；极低分全部打回重新设计场景脚本，超过2次打回则升级为 CRITICAL。

---

### 4.5 Portfolio 管理机制

以投资组合的思维管理多本书并行，核心逻辑是数据驱动的资源动态分配。

**生命周期阶段**：规划中 → 冷启动（0-20章）→ 验证期（20-50章）→ 增长期（50章+）→ 已停更

**核心决策阈值**：
- 留存率 < 25%：停更，释放资源
- 留存率 25-40%：优化，调整内容策略
- 留存率 > 40%：加仓，增加更新频率

**并行规模**：MVP 支持3本并行，完整方案支持10本以上。

## 七、技术需求清单

> 以下为实现上述业务架构所需的关键技术要求，不含具体实现方案，在各模块开发文档中展开。

### LLM 相关
- [ ] LLM 统一路由层，支持系统级/Agent级/任务级三层优先级配置
- [ ] 支持多厂商适配：DeepSeek、Anthropic、OpenAI、Ollama（本地）
- [ ] 主模型失败时自动切换 fallback 模型，不中断任务
- [ ] 配置文件热加载（修改模型配置不需要重启服务）
- [ ] LLM 调用成本追踪，按 Agent 和模型维度汇总

### Agent 相关
- [ ] Agent 流水线编排，支持串行、条件分支、循环
- [ ] RevisionRequest 事件总线，Agent 间解耦通信
- [ ] 多版本并行生成与投票评分机制
- [ ] 每章生成失败自动重试，超过阈值升级处理
- [ ] Agent 任务执行记录，支持问题复现和调试

### 记忆系统相关
- [ ] 三层记忆的存储与读取，Layer2 每章自动更新
- [ ] Layer2 每30章从原文自动重建（防漂移）
- [ ] B5b ContextAssembler 按章节类型动态调整三层配比
- [ ] 三层记忆总 token 量上限控制

### 平台接入相关
- [ ] 浏览器插件（Chrome Extension）开发
- [ ] 插件拦截平台 XHR 请求获取章节完读率
- [ ] 插件自动填充发布内容，保留人工确认步骤
- [ ] 插件与后端的安全通信（专用 API Key）
- [ ] 番茄小说后台适配（MVP），七猫后台适配（完整方案）

### 调度与并行相关
- [ ] 多书并行任务调度，互不阻塞
- [ ] 每日定时任务：数据采集提醒、策略分析、章节生成
- [ ] Portfolio 层面的资源分配逻辑（哪些书优先生成）
- [ ] CRITICAL 级别事件的人工通知机制

### 数据相关
- [ ] 章节段评情感分析，提取关键问题标签
- [ ] 书级和章节级数据的趋势计算
- [ ] 跨书数据对比（完整方案，A4使用）
