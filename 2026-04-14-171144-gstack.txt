     目录规范设计计划

     Context

     用户管理多个 AI 驱动的软件项目（当前有
     hermes、consilium），需要一套规范的文档目录结构，满足：
     1. 全局开发规范：编码规范、TDD 流程等对所有项目生效
     2. 阶段隔离：每个开发阶段只加载对应文档，不加载其他阶段文档，减少 context
     消耗
     3. TDD 工作流：先从接口文档写测试用例，再写业务代码
     4. 迭代支持：项目上线后的迭代需求有独立的文档组织

     核心设计原则

     1. 阶段隔离（减少 context 消耗）

     PROGRESS.md 是项目进度的唯一事实来源，记录所有阶段的完成状态：

     # Hermes 项目进度

     ## 当前状态
     - **阶段**：03 - Module Design（进行中）
     - **迭代**：v1.0（首次发布，尚未上线）

     ## 阶段进度
     | 阶段 | 名称 | 状态 | 完成时间 |
     |------|------|------|---------|
     | 01 | Requirements | ✅ 完成 | 2026-03-10 |
     | 02 | System Design | ✅ 完成 | 2026-03-25 |
     | 03 | Module Design | 🔄 进行中 | — |
     | 04 | Development | ⏳ 待开始 | — |
     | 05 | Testing | ⏳ 待开始 | — |
     | 06 | Deployment | ⏳ 待开始 | — |
     | 07 | Operations | ⏳ 待开始 | — |

     ## 迭代历史
     | 版本 | 状态 | 发布时间 |
     |------|------|---------|
     | v1.0 | 🔄 开发中 | — |

     CLAUDE.md 只做引用控制，不记录进度。每次只指向当前阶段：

     # Project: Hermes
     > 当前进度见 PROGRESS.md

     ## 必读（全局规范）
     - @../../standards/coding-standards.md
     - @../../standards/tdd-workflow.md

     ## 当前阶段文档（仅此阶段）
     - @./stages/03-module-design/module-design.md
     - @./stages/03-module-design/api-spec.md

     ## 其他阶段文档（勿主动读取）
     - stages/01-requirements/、stages/02-system-design/ 已完成，按需查阅

     进入下一阶段时，只需更新两处：PROGRESS.md 的状态行 + CLAUDE.md 的引用路径。

     2. TDD 工作流（第 04 阶段专用）

     stages/04-development/WORKFLOW.md 规定严格的两步流程：

     步骤 1：写测试（Red Phase）
       - 读取 ../03-module-design/api-spec.md
       - 在 ./tests/ 下为每个模块创建测试文件
       - 运行测试 → 全部应该 FAIL（尚无实现）
       - ⛔ 测试文件写入后即锁定，禁止任何修改

     步骤 2：写实现（Green Phase）
       - 根据测试文件编写业务代码（只能改实现，不能改测试）
       - 运行测试 → 全部应该 PASS
       - Refactor → 保持测试通过
       - 如果测试无法通过，修改实现而非测试

     测试文件不可修改原则：
     测试用例从 api-spec.md 派生，代表接口契约。一旦写入
     tests/，任何修改都意味着"降低标准"，严格禁止。若接口本身需要变更，应先修改
     api-spec.md 并经过评审，再重新生成测试。

     在 CLAUDE.md 中通过指令强制此约束：

     ## ⛔ 禁止修改
     - stages/04-development/tests/ 下的所有文件为只读契约
     - 不得以任何理由修改测试文件（包括"让测试通过"）
     - 如需变更，走接口变更流程：修改 api-spec.md → 重新评审 → 重写测试

     CLAUDE.md 在阶段 04 时引用：
     - ../03-module-design/api-spec.md（测试用例的输入，参考用）
     - ./WORKFLOW.md（TDD 流程）
     - ./tests/（当前测试用例状态，只读）
     - 不引用任何业务实现代码文档（让模型自由实现）

     3. 迭代管理（上线后）

     每次迭代在 iterations/vX.Y/ 下创建独立目录，有自己的 CLAUDE.md：

     # Iteration: v1.1
     **Base Version:** v1.0 (production)

     ## 变更范围
     - @./requirements.md      # 本次迭代需求
     - @./api-changes.md       # 接口变更（diff 格式）

     ## 参考（仅在需要时手动读取）
     - ../../stages/03-module-design/api-spec.md   # 原始接口（对比用）

     ## 工作流
     与 Stage 04 相同：先写 ./tests/ 中的新测试，再实现变更

● 先说总体感受：整体设计思路清晰，阶段隔离 + TDD 锁定的思路是对的。但要真正驱动
  模型"独立交付生产级代码"，有几个关键薄弱点，按影响力排序：

  🔴 Finding 1：软约束 vs 硬约束（最关键）

  当前 CLAUDE.md 里的"禁读其他阶段文档""测试文件只读"全是文本说服。模型在压力场
  景下（比如测试一直失败）可能"创造性"绕过 —— 去改测试而非改实现。

  建议：把软约束变成机械阻断
  - 用 .claude/settings.json 的 permissions.deny：进入阶段 04 后，阻止
  Edit/Write 对 stages/04-development/tests/** 的写入
  - 用 pre-commit hook 检测 tests/ 目录变更，遇到则 reject
  - 用 chmod -w + git hook 双重保护

  🔴 Finding 2：接口文档可执行化

  api-spec.md 是自然语言
  Markdown，模型从中"解读"出测试。每次解读可能不一致，甚至会漏掉边界情况。这是
  TDD 链条最脆弱的一环。

  建议：接口源头用可执行 schema
  - Pydantic models（Python 原生） 或 OpenAPI YAML 作为真实源头
  - api-spec.md 只做设计说明，不作为测试派生源
  - 测试从 schema 机械生成（或至少有脚本校验测试覆盖了所有 schema 字段）

  这样消除"人解读文档 → 模型解读文档"的双重模糊。

  🟡 Finding 3："生产级"维度缺失

  TDD 测试通过 = 功能正确，但生产级 = 功能正确 + 可观测性 + 性能 + 安全 +
  可恢复。当前流程里这些要素要么散落在阶段 06/07（太晚），要么根本没提。

  建议：把"非功能性要求"前置到阶段 03 的 api-spec

  每个接口强制填写：
  - SLO：P99 响应时间、可用性目标
  - 日志字段：必须打印的 trace_id、业务字段
  - 指标：暴露的 Prometheus 指标
  - 降级策略：外部依赖挂掉时的 fallback
  - 安全：认证方式、限流阈值

  阶段 04 的 TDD 除了功能测试，还要有性能测试 + 日志字段断言。

  🟡 Finding 4：阶段推进缺少门禁

     目录规范硬化升级计划

     Context

     基础目录结构已落地（standards/、stages/、PROGRESS.md、CLAUDE.md 阶段指针已
     就位）。但该结构靠文本说服模型遵守规则，在真实生产级交付场景下存在 4
     个脆弱点：

     1. 软约束：CLAUDE.md
     的"禁读其他阶段""测试只读"全靠模型自律，模型在压力下可能绕过
     2. 接口文档模糊：api-spec.md 是
     Markdown，模型每次解读可能不一致，测试生成不可复现
     3. 生产级维度缺失：当前流程只保证功能正确，没有要求 SLO/日志/监控/降级
     4. 阶段推进无门禁：PROGRESS.md 手动维护，没有"Definition of
     Done"校验，可以在未完成时假装完成

     新增需求：每阶段规定确定的产出清单，产出完成后进度自动维护为完成（减少人工
     维护，让状态与事实一致）。

     目标：把当前的"纸面规范"升级为"机械可验证、自动推进、难以绕过"的工作流，真
     正驱动模型独立交付生产级代码。

     from pydantic import BaseModel, Field
     from datetime import date
     from typing import Literal

     class FetchDailyRequest(BaseModel):
         symbol: str = Field(..., pattern=r"^\d{6}\.(SZ|SH)$",
     description="股票代码")
         start_date: date
         end_date: date
         adjust: Literal["qfq", "hfq", "none"] = "qfq"

     class DailyBar(BaseModel):
         trade_date: date
         open: float = Field(..., gt=0)
         high: float = Field(..., gt=0)
         low: float = Field(..., gt=0)
         close: float = Field(..., gt=0)
         volume: int = Field(..., ge=0)

     class FetchDailyResponse(BaseModel):
         symbol: str
         bars: list[DailyBar]

     # 异常声明（编码在类型里，不在注释里）
     class DataLayerErrors:
         INVALID_SYMBOL = "data_layer.invalid_symbol"
         DATA_NOT_FOUND = "data_layer.data_not_found"
         UPSTREAM_TIMEOUT = "data_layer.upstream_timeout"

     ## 接口：FetchDaily

     ### 功能契约（Schema）
     参见 `schemas/data_layer.py::FetchDailyRequest / FetchDailyResponse`

     ### SLO 契约
     | 指标 | 目标 |
     |------|------|
     | P50 延迟 | < 50ms |
     | P99 延迟 | < 300ms |
     | 可用性 | 99.5% |
     | 错误率 | < 0.1% |

     ### 可观测性契约
     **必须打印的日志字段：**
     - `trace_id`（贯穿整条请求链路）
     - `symbol`, `start_date`, `end_date`（业务参数）
     - `duration_ms`（性能追踪）
     - `upstream_source`（Tushare/AkShare，哪个数据源命中）

     **必须暴露的指标（Prometheus）：**
     - `hermes_data_fetch_requests_total{symbol, source, status}` (Counter)
     - `hermes_data_fetch_duration_seconds{symbol, source}` (Histogram)
     - `hermes_data_fetch_cache_hit_total{symbol}` (Counter)

     ### 降级契约
     | 场景 | 降级策略 |
     |------|---------|
     | Tushare 超时 | 自动切换 AkShare |
     | AkShare 也失败 | 返回最近缓存（带 stale 标记）|
     | 缓存也没有 | 返回 UPSTREAM_TIMEOUT 错误，记录到补跑队列 |

     ### 安全契约
     - 认证：内部调用，无需用户认证
     - 限流：单实例 10 req/s（防止上游封禁）
     - 输入校验：symbol 正则匹配，date 范围 ≤ 3 年

     核心脚本：

     1. scripts/check-stage.sh <stage-id>
       - 读取对应阶段的 deliverables.yaml
       - 逐项验证每个 deliverable
       - 输出：PASS / FAIL + 详细缺失项
       - 返回码：0 = 全部通过，非 0 = 有缺失
     2. scripts/update-progress.sh <project>
       - 对每个阶段运行 check-stage.sh
       - 根据结果自动更新 <project>/PROGRESS.md：
           - 全部通过 → 该阶段标记 ✅ 完成（含时间戳）
         - 部分通过 → 🔄 进行中（附进度百分比）
         - 全部失败 → ⏳ 待开始
       - 只读取事实（文件 + 命令），不修改 CLAUDE.md 或做推进
       - 可以作为 pre-commit hook 或 CI 步骤自动运行
     3. scripts/advance-stage.sh <project> <from-stage> <to-stage>
       - 先运行 check-stage.sh <from-stage>，必须全部通过
       - 更新 PROGRESS.md：from-stage 标记为 ✅，to-stage 标记为 🔄
       - 更新 <project>/CLAUDE.md：
           - 把 "当前阶段" 指针指向 to-stage
         - 替换"当前阶段文档"区块的文件引用
       - 创建 git commit：stage: advance <project> from <from> to <to>
       - 如果推进到阶段 04，自动执行 lock-tests.sh

     自动进度维护触发时机：
     - Git pre-commit hook：每次提交时跑 update-progress.sh，PROGRESS.md
     自动更新
     - 手动触发：开发过程中可随时 make progress 查看当前状态
     - CI 跑：推送后跑一遍，保证仓库状态和事实一致
